# v2 llms.txt Research

Date: 2026-05-22. Budget: live WebSearch + fetch via context-mode.

## 1. llms.txt spec (May 2026)

- Canonical: https://llmstxt.org (spec maintained at AnswerDotAI/llms-txt).
- Format unchanged from original Sept-2024 proposal — still markdown-based, root-hosted at `/llms.txt`.
- Required structure (strict order):
  1. H1 — project/site name (only required element).
  2. Blockquote — short summary.
  3. Zero+ non-heading markdown sections (context/prose).
  4. Zero+ H2 sections containing markdown lists of `[name](url): description` links. Special H2 `## Optional` = lower-priority content the parser may drop on token pressure.
- `llms-full.txt` is a **community convention, not in the spec** — single file with every page expanded, used by Vercel, Anthropic, Cloudflare, FastAPI etc.
- Parsing tool: `llms_txt` Python pkg (FastHTML/AnswerAI), gives `parse_llms_file` → structured dict. No standalone "validator" CLI; structural validity is whatever `llms_txt.parse_llms_file` accepts.
- Sources: https://llmstxt.org , https://github.com/AnswerDotAI/llms-txt

## 2. mkdocs-llmstxt plugin

- Latest: **0.5.0** (PyPI, 2025-11-20). Author: pawamoy.
- Install: `pip install mkdocs-llmstxt`.
- Requires `site_url` set in `mkdocs.yml`.
- Config surface (verified from docs):
  ```yaml
  plugins:
    - llmstxt:
        markdown_description: "Long description of the project."
        full_output: llms-full.txt          # emit llms-full.txt too
        autoclean: true                     # BeautifulSoup HTML cleanup
        preprocess: tools/llmstxt_clean.py  # optional Python hook
        sections:
          Tutorials:
            - tutorials/*.md
          How-to:
            - how-to/*.md
          Reference:
            - reference/**/*.md
          Explanation:
            - explanation/*.md
  ```
- Per-file description: `- path.md: "Description"`; otherwise plugin derives title from page H1.
- **Does NOT auto-read frontmatter `summary:`.** Description must be supplied inline in `sections:` OR injected via the `preprocess` hook. Implication: we need a small build-time generator (Python) that walks `docs/`, reads frontmatter, and produces the `sections:` mapping fed into mkdocs.yml (or writes llms.txt directly and skips the plugin's TOC generation).
- Markdown body for each page is generated by: render HTML → BeautifulSoup clean → Markdownify. Dynamic content (macros, includes) is preserved because it goes through the rendered HTML.
- Sources: https://pawamoy.github.io/mkdocs-llmstxt/ , https://pypi.org/project/mkdocs-llmstxt/ 0.5.0

## 3. Sizing

- Current corpus: ~700 pages. v2 contract caps `summary` at 200 chars.
- `llms.txt` (TOC + summaries): 700 × (~80 char title+url + 200 char summary + markup) ≈ 700 × 320 B ≈ **~220 KB**. Well under 1 MB. Single file across all four Diátaxis quadrants is fine.
- `llms-full.txt`: assume mean page body 6 KB after autoclean (typical Diátaxis page, code + prose). 700 × 6 KB ≈ **~4.2 MB**. Exceeds 1 MB single-file budget.
- Recommended split:
  - `/llms.txt` — global TOC + summaries, all quadrants. ~220 KB.
  - `/llms-full.txt` — full body, **chunked by Diátaxis quadrant**:
    - `/llms-full-tutorials.txt`
    - `/llms-full-how-to.txt`
    - `/llms-full-reference.txt` (largest; may need further split by component if >1 MB — e.g. `runner`, `framework`, `mocker`)
    - `/llms-full-explanation.txt`
  - Each ≤1 MB; agents pick by quadrant. `llms.txt` references all four full bundles in a `## Full content` H2.

## 4. Frontmatter source mapping

v2 contract §6 keys → llms.txt usage:

| Frontmatter | Used as |
|---|---|
| `id` | stable anchor / dedup key |
| `type` | grouping into Diátaxis H2 section |
| `summary` | description after `:` in link line |
| `keywords` | optional — append in parentheses for retrieval boost (not part of spec; harmless) |
| `status` | filter — `deprecated` entries go under `## Optional` |
| `applies_to` | optional grouping for component bundles |

Generator: `tools/build_llmstxt.py` — walks `docs/`, reads frontmatter via `python-frontmatter`, emits `llms.txt` directly. Bypasses the plugin's `sections:` mapping (cleaner than templating mkdocs.yml). Plugin still used for `llms-full.txt` autoclean-to-markdown pipeline.

## 5. Hosting

- Lives at site root: `https://docs.qaas/llms.txt` and `https://docs.qaas/llms-full*.txt`.
- MkDocs Material serves any file in `docs/` (or generated into `site/`) as-is. Plugin writes into `site/` during `mkdocs build`.
- MIME: served as `text/plain` by default static hosts (GitHub Pages, S3, nginx auto-detect by `.txt`). No special config needed. Verify with `curl -I` post-deploy.

## 6. Quality gate

- No official `llms-txt-validator` CLI exists (May 2026).
- Verification approach:
  1. `python -c "from llms_txt import parse_llms_file; parse_llms_file(open('site/llms.txt').read())"` — exits non-zero on structural break.
  2. Custom assertions in `tools/ci/check-llmstxt.sh`:
     - H1 present, exactly one.
     - Blockquote present after H1.
     - Every link line parses to `[name](url): description`.
     - Total bytes ≤ 1 MB (warn at 800 KB).
     - Every `id` in frontmatter appears once.
     - `llms-full*.txt` each ≤ 1 MB.
  3. Optional: `htmltest`-style URL check that every link resolves on built site.

## 7. Cross-reference v2 contract

- Contract §6: `summary: "<= 200 chars"`. `tools/ci/check-frontmatter.sh` already enforces (per session memory of contract revision 2026-05-22 dropping `ai_summary`).
- Verify `check-frontmatter.sh` rules:
  - `summary` required, ≤200 chars, no newlines, no markdown links inside.
  - Reject legacy `ai_summary` key.
- Result: llms.txt entries inherit guaranteed-clean one-line summaries. No re-validation needed inside the llms.txt generator.

## 8. Kiwix MCP / ZIM integration

- Yes — embed `llms.txt` and `llms-full-*.txt` inside the ZIM.
- ZIM packages anything in the `site/` tree; `zimit`/`mwoffliner`/`sotoki`-style tooling preserves arbitrary `.txt` files at their URL paths.
- Kiwix MCP `search` returns text content of stored articles; agents can `get` the `A/llms.txt` entry directly without an external HTTP fetch.
- Recommended: in the Kiwix MCP server config, expose `llms.txt` as a "manifest" resource (resource URI `kiwix://qaas-docs/llms.txt`) so MCP clients discover the corpus map in one call.
- Bonus: ZIM full-text index already covers `llms-full-*.txt`, so per-quadrant BM25 search comes free.

## Recommendations (summary)

1. Adopt `mkdocs-llmstxt` 0.5.0 for the autoclean→markdown pipeline (`full_output`).
2. Write `tools/build_llmstxt.py` to generate `llms.txt` from frontmatter (plugin's `sections:` does not read frontmatter).
3. Emit one `llms.txt` + four `llms-full-<quadrant>.txt` bundles; split reference further if >1 MB.
4. CI gate: `python -m llms_txt.parse` + custom assertions in `tools/ci/check-llmstxt.sh`.
5. Embed all `.txt` artifacts in the ZIM; expose `llms.txt` as a Kiwix MCP resource.
6. No new frontmatter keys needed — `summary`/`id`/`type`/`keywords`/`status` already cover it.
